# 管理员用户迁移操作手册

本文档面向 PrivHub 普通管理员，说明如何发起迁移、追加用户、发送迁移指引、导出数据、确认迁移完成，以及导入迁移用户。

## 功能入口

管理员登录后进入：

```text
/console/user-migration
```

侧边栏路径：

```text
管理员 -> 用户迁移
```

该页面包含：

- 迁移任务
- 追加与目标
- 导出确认
- 导入用户
- 表达式文档

## 推荐迁移生命周期

1. 在“表达式文档”中确认可用字段和示例。
2. 在“迁移任务”中填写筛选表达式并预览命中用户。
3. 创建迁移任务，按需勾选“创建后立即发送迁移邮件”。
4. 在“追加与目标”中查看目标用户，必要时追加单个 CAH/user_id 或追加新的表达式命中用户。
5. 用户通过邮件中的 `/migrate/...` 链接登录指定原账户并确认迁移。
6. 在“导出确认”中获取最新待导出 JSON。
7. 接收方导入成功后，管理员点击“确认已迁移”。
8. 系统自动以“已迁移”为原因禁用原账户，并使原 Web 会话失效。

## 筛选表达式

表达式参考 Cloudflare 防火墙表达式风格，但只开放用户迁移白名单字段。

常用字段：

| 字段 | 含义 |
| --- | --- |
| `username` | 用户名 |
| `email` | 用户当前邮箱 |
| `cah` / `cah_id` | Customer Account Handle |
| `provider` | 第三方登录方式或 password |
| `status` | 用户状态 |
| `role` | 用户角色 |
| `group` | 用户分组 |

常用运算符：

| 运算符 | 含义 |
| --- | --- |
| `==` / `eq` | 等于 |
| `!=` / `ne` | 不等于 |
| `contains` | 包含 |
| `matches` | 正则匹配 |
| `in` | 集合匹配 |
| `and` / `or` / `not` | 逻辑组合 |

示例：

```text
status == "enabled" and email matches ".*@example\\.com$"
provider in {"github", "oidc"} and group == "paid"
(role == "common" or role == "support") and not email contains "+test"
```

状态值：

- `enabled` 或 `1`
- `disabled` 或 `2`

角色值：

- `guest` 或 `0`
- `common` 或 `1`
- `support` 或 `5`
- `admin` 或 `10`
- `root` 或 `100`

注意：

- 超级管理员不会被加入迁移目标。
- 表达式筛选默认只包含未软删除且启用的用户；如果表达式显式使用 `status` 字段，则按表达式状态匹配。
- 表达式为空时不会创建批量目标。
- 管理员应先预览命中数量，再创建迁移。

## 创建迁移任务

在“迁移任务”中填写：

- 名称：内部识别用，例如 `2026 Q2 paid users migration`
- 说明：可选
- 筛选表达式：目标用户规则
- 创建后立即发送迁移邮件：按需勾选

创建后系统会生成迁移 ID，例如：

```text
mig_xxxxxxxxxxxxxxxx
```

迁移链接格式：

```text
/migrate/<migrate_id>/#~/t/<migrate_token>/<access_token>/~/u/<user_token>
```

token 自动生成，明文只用于链接生成和邮件发送，不在数据库中明文保存。`user_token` 是必填安全参数，缺失时无法验证或提交迁移。

## 追加目标用户

进入“追加与目标”，选择当前迁移任务。

可以使用两种方式追加：

### 通过表达式追加

填写新表达式，例如：

```text
status == "enabled" and group == "vip"
```

系统会跳过已在该迁移任务中的重复用户。

### 通过用户追加

填写以下任意用户标识：

- `user_id`
- `CAH`

同时填写迁移通知发送邮箱。该邮箱优先于用户档案邮箱。

## 邮件发送规则

迁移邮件是事务邮件，使用 Postmark transactional `outbound` stream。

发送规则：

- 创建迁移或追加目标时可立即发送。
- 每个目标用户记录独立的发送状态。
- 批量发送中部分失败不会被视为整体成功。
- 邮件失败原因会写入目标记录。
- 重复提交使用幂等键降低重复投递风险。

迁移邮件会要求用户：

- 打开 `/migrate/...` 链接。
- 登录指定 CAH 的原 PrivHub 账户。
- 完成迁移确认。

## 用户确认流程

用户打开链接后，全流程页面和 API 都位于 `/migrate` 前缀下。

流程：

1. 页面验证迁移 token。
2. 如果当前浏览器已经登录指定账户，可直接确认。
3. 如果未登录，页面会要求输入原 PrivHub 用户名和密码。
4. 登录账户必须与迁移目标一致。
5. 成功后系统记录迁移数据，目标状态变为 `captured`。

## 导出数据

进入“导出确认”。

页面列出已经由用户确认、但尚未被接收方确认导入的迁移数据。

管理员可以：

- 下载单个 JSON。
- 将 JSON 提供给接收方。
- 等接收方确认导入成功后点击“确认已迁移”。

排序规则：

- 按迁移目标记录创建时间从早到晚获取最新待处理数据。

列表只展示元数据、数据大小和下载地址，不直接返回完整迁移 JSON。下载尝试不代表接收方已成功取得文件；只有接收方确认导入成功后，才执行“确认已迁移”。

## 确认已迁移

只有接收方明确表示导入成功后，才点击“确认已迁移”。

确认后系统会：

- 将迁移目标状态设为 `migrated`。
- 将原用户状态设为禁用。
- 禁用原因写为 `已迁移`。
- 递增原用户 Web 会话版本，使旧 Web 会话失效。

该操作不可当作“仅导出”使用。

## 导入用户

进入“导入用户”。

填写：

- CAH
- 邮箱
- 迁移 JSON 数据

系统会拒绝以下情况：

- CAH 已存在。
- 邮箱已存在。
- CAH 格式无效。
- JSON 非法。

导入成功后：

- 系统生成内部用户名。
- 创建新用户。
- 账户初始禁用，原因是等待迁移密码设置。
- 发送 `/migrate/import/#~/i/...` 设置密码邮件。
- 用户设置密码后账户启用。

如果密码设置邮件发送失败，导入本身仍然成功，记录状态会显示为 `email_failed`，管理员可以使用返回的设置链接进行恢复处理。此时不要重复导入同一 CAH 或邮箱，重复导入会被拒绝。

## 导出和导入的数据范围

导出包含：

- CAH
- 管理员指定的迁移邮箱
- 展示名、分组、额度、用量计数、设置、备注、Stripe customer、订阅数据等账户字段
- 充值记录
- 充值优惠券
- 兑换码元数据，不含兑换码密钥
- API token 元数据，不含 token key
- 使用限制窗口计数
- 站内消息快照

导出不包含：

- 用户 ID
- 用户名
- 密码
- 第三方登录 ID
- access token
- API key 明文
- session
- passkey
- 2FA
- 使用日志
- 任务/绘图日志
- 管理员审计日志

导入会实际恢复安全可落库的数据，例如账户字段、充值记录、充值优惠券和使用限制窗口。无法安全恢复或不应恢复的项目会保留在迁移 JSON 中作为历史快照。

## 风险控制建议

- 先用小范围表达式测试完整流程。
- 每次批量迁移前先预览命中数量。
- 邮件失败时先修复邮箱或 Postmark 配置，再追加或重新生成目标。
- 不要在接收方未确认导入成功前点击“确认已迁移”。
- 对管理员账户迁移保持谨慎，超级管理员不会被自动加入目标。
- 将下载的迁移 JSON 当作敏感数据处理，避免放入公开工单、聊天或日志。
